## B. NL → 标准化指令：指令规范化模块（预计 1-1.5 周）

> 目标：补齐"原始模糊 NL → 标准无歧义指令"链路
> 对应工程清单 Layer 2.2 / 论文v4.6中"指令明确无误时直接形成标准化任务计划，若有指令模糊…由LLM推理补充、消除歧义"
> 采纳 GPT 方案的核心设计，但简化实现粒度

### B.1 新增 json_extraction.py

```
文件：agents/json_extraction.py

功能：
  从 LLM 原始输出中提取 JSON 对象
  支持 fenced ```json``` 块 / 裸 JSON / 失败返回
  与 code_extraction.py 风格统一

接口：
  extract_json_object(raw_response: str) -> JsonExtractionResult
  JsonExtractionResult: { ok, data, method, error }

验收：
  3个测试用例：fenced / bare / invalid
```

### B.2 新增 InstructionNormalizerAgent

```
文件：agents/instruction_normalizer_agent.py

功能：
  输入原始 NL 指令 → 输出结构化规范化结果
  复用 LLMClient，风格对齐 PlannerAgent

输出格式（简化版，不含 GPT 方案中的 evidence 字段）：
  {
    "status": "complete" | "incomplete",
    "standard_instruction": "..." | null,
    "resolved_fields": { segment_id, assigned_actors, tasks, relations, constraints },
    "missing_fields": [...],
    "ambiguities": [{ "span": "...", "reason": "...", "field": "..." }]
  }

关键规则：
  不得编造 actor/target/deadline/resource
  complete 仅在必要字段齐全时使用
  incomplete 时 standard_instruction 必须为 null

验收：
  能用 --limit 1 跑通一条模糊指令
```

### B.3 新增规范化 prompt

```
文件：prompts/instruction_normalization_prompt.md

内容：
  角色定义 + 输出 JSON schema + 2 个示例（1 complete + 1 incomplete）
  强调"不得脑补缺失信息"

验收：
  prompt 渲染后 token 数 ≤ 2000
```

### B.4 新增模糊指令数据集（第一版 10 条）

```
文件：datasets/phase1_ambiguous_nl_cases.jsonl

样本格式：
  {
    "sample_id": "1f_001_...",
    "raw_instruction": "...",
    "expected_status": "complete" | "incomplete",
    "expected_missing_fields": [...],
    "expected_ambiguity_spans": [...],
    "tags": [...]
  }

覆盖分布：

验收：

```

## E. NL 全串联单 Agent 原型（预计 1 周）

> 目标：raw NL → standard NL → GCJP → verify 单 Agent 端到端跑通
> 对应工程清单 Layer 3

### E.1 串联管道实现

```
文件：experiments/exp_03_single_agent_nl_pipeline.py

流程：
  raw NL → InstructionNormalizerAgent →
  if complete → PlannerAgent → GCJP → verify → report
  if incomplete → 记录，不继续

数据源：
  B.4 的 complete 样本 + D.2 种子数据集中带 NL 指令的样本

指标：
  end_to_end_z3_pass_rate
  normalization_error_rate（NL理解错导致的下游失败）
  code_generation_error_rate（代码生成本身的失败）
  失败归因分布

验收：
  强模型端到端 Z3 通过率 ≥ 60%（M-2 通过条件）
```

### E.2 决策点

```
如果通过率 ≥ 60%：
  进入 Layer 4 种子数据集大规模扩展 + Layer 5 多Agent

如果 40% ≤ 通过率 < 60%：
  分析失败归因，针对性增强 prompt 或简化样本

如果通过率 < 40%：
  回退到 JSON→确定性翻译 模式
  或降低样本复杂度到 Level 1-2
```

## 与 GPT 方案的差异说明

| 方面 | GPT 方案 | 本清单 |
|------|---------|-------|
| 定位 | 阶段 1 补全 | 工程 Layer 2-3 的自然推进 |
| 数据集 | 第一版 12 条，覆盖 12 种模糊类型 | 第一版 10 条，覆盖 7 种，降低粒度 |
| evidence 字段 | 要求模型输出每个解析字段的来源片段 | 暂不要求，后续微调阶段再加 |
| hallucination 检测 | 用 false_complete_rate + evidence 交叉验证 | 仅用 false_complete_rate，规则判断 |
| 非 live demo | 独立 demo 文件 | 统一到一个 nonlive regression |
| 文档编号 | 新增 1F/1G 但不改已有编号 | 同意此策略 |
| 修复闭环接入 | 明确"先不接入" | 同意，E 阶段跑通后再考虑 |
| JSON→GCJP 对比 | 未涉及 | 本清单新增为 C 部分 |
| 种子数据集 | 未涉及 | 本清单新增为 D 部分 |
