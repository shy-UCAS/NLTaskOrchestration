## 阶段 0：UNSAT 归因降噪（最小版本）

> **前置于修复闭环，确保反馈信息质量**
>
> 预计周期：2-3天

### 0.1 问题背景

当前 Z3 的 `unsat_core` 会混入框架级辅助约束（`start_nonneg_*`、`end_def_*`、`dur_lb_*`、`dur_ub_*`），这些对 Z3 求解是必要的，但不适合作为反馈给 LLM 修复的主要错误原因。如果不做降噪，阶段 2 的修复实验在 UNSAT 场景下会因反馈噪声太大而效果失真，无法判断是模型能力不足还是反馈信息不够。

### 0.2 需要修改的文件

```
gcjp/constraint_templates.py   # 在 solve_and_report() 中增加过滤逻辑
```

### 0.3 实现方案

在 `solve_and_report()` 返回的结果中增加两个字段：

```python
{
    "unsat_core_raw": [
        "start_nonneg_T1",
        "dur_lb_T1",
        "end_def_T1",
        "time_window_auto_T1_latest",
        "dep_T1__T2"
    ],
    "unsat_core_semantic": [
        "time_window_auto_T1_latest",
        "dep_T1__T2"
    ]
}
```

过滤规则：

- 移除：`start_nonneg_*`、`end_def_*`、`dur_lb_*`、`dur_ub_*`
- 保留：`time_order_*`、`time_window_*`、`sync_*`、`group_sync_*`、`resource_*`、`capability_*`、`physical_feasibility_*`、用户自定义 `source_label`

### 0.4 增加归因翻译（最小版本）

为每个保留的 unsat core 标签生成一句自然语言建议，基于规则模板而非 LLM：

```python
ATTRIBUTION_TEMPLATES = {
    "time_window": "任务 {task_id} 的截止时间约束过紧，建议放宽 deadline 或减少前置任务",
    "time_order": "任务 {before} → {after} 的顺序依赖参与了冲突，考虑是否可以改为 parallel",
    "resource": "资源 {resource_type} 总消耗超出上限 {max_value}，建议减少任务或增大配额",
    "capability": "任务 {task_id} 要求能力 {required} 但执行者只有 {actual}，任务无法分配",
    "physical_feasibility": "任务 {task_id} 物理最短耗时 {min_duration} 超出可用时间窗",
}
```

### 0.5 验证方式

在现有 demo_02/05/10/11 的 UNSAT 场景上跑一遍，确认 `unsat_core_semantic` 不含框架约束，且归因翻译可读。

> **最高优先级，验证核心研究假设**
>
> 预计周期：1-2周

### 1.1 需要新增的文件

```
agents/

prompts/

experiments/

demos/

```

### 1.2 输入格式

第一步使用结构化子任务描述，跳过 NL→结构化：

```json
{
  "segment_id": "seg_recon_strike",
  "assigned_actors": ["fleet_1"],
  "task_description": "fleet_1 first recon hq_1, then strike hq_2",
  "available_actions": ["reconnaissance", "strike"],
  "capability_model": "configs/capability_model.yaml",
  "environment_config": "configs/environment_config.yaml"

```

### 1.3 Prompt 设计要点

**使用 3-4 个 few-shot 示例**（而非单个示例），覆盖不同任务模式：

| 示例编号 | 覆盖模式 | 来源 |
|---------|---------|-----|
| 示例 1 | solo段 + sequence 依赖 | demo_03/06 |
| 示例 2 | 多 actor + parallel 关系 | demo_08 |
| 示例 3 | sync 会合 + barrier | demo_09 |
| 示例 4 | 带 physical_feasibility 的复杂段 | demo_05 |

Prompt 结构：

```
[System] 你是 GCJP v1 代码生成器。只输出 Python GCJP 代码，不输出解释。

[API 规范] GCJP v1 白名单方法列表 + 参数签名 + 关系类型 + 约束类型

[示例 1] 输入描述 → 正确 GCJP 代码
[示例 2] 输入描述 → 正确 GCJP 代码
[示例 3] 输入描述 → 正确 GCJP 代码
[示例 4] 输入描述 → 正确 GCJP 代码

[当前任务] {task_description}
```

注意 token 预算：API 规范 + 4 个示例可能占 3000-5000 tokens，需要实测确认不会挤压生成空间。如果超出预算，优先保留示例 1 和示例 4，裁剪 API 规范到关键方法。

### 1.4 评价指标

| 指标 | 含义 |
|------|------|
| `safety_pass_rate` | AST 安全检查通过率 |
| `execution_success_rate` | GCJP 沙箱执行成功率 |
| `builtgraph_success_rate` | BuiltGraph 生成成功率 |
| `z3_sat_rate` | Z3 约束验证通过率（仅对应 SAT 的样例） |
| `first_pass_rate` | 一次生成直接通过全部验证的比率 |
| `constraint_completeness_rate` | 生成的约束集合与 reference 的召回率（防止假阳性 SAT） |
| `error_type_distribution` | 失败时的错误类型分布 |

`constraint_completeness_rate` 的计算方式：以手写的正确 GCJP（或 JSON loader 输出的 BuiltGraph）为 reference，统计 LLM 生成的 BuiltGraph 中约束的召回率。如果 LLM 漏掉了应有的 resource_constraint 或 physical_feasibility_constraint，即使 Z3 报 SAT 也应标记为不完整。

### 1.5 决策点

- 如果简单样例 `first_pass_rate` ≥ 50%：进入阶段 2
- 如果 `first_pass_rate` < 30% 但 `safety_pass_rate` ≥ 80%：说明代码结构对但约束有误，考虑把约束生成改为"LLM 填模板参数"模式
- 如果 `safety_pass_rate` < 50%：说明 API 设计过于复杂或 prompt 不足，需要简化 API 或增加示例

## 阶段 2：LLM 修复闭环原型

> **验证结构化错误反馈是否有效**
>
> 预计周期：1-2周
>
> 前置依赖：阶段 0（归因降噪）+ 阶段 1

### 2.1 需要新增的文件

```
agents/
  repair_agent.py             # 修复 Agent 封装

prompts/
  gcjp_repair_prompt.md       # 修复 prompt 模板

experiments/
  exp_02_gcjp_repair_loop.py

demos/
  demo_15_gcjp_repair_loop.py
```

### 2.2 修复流程

```
LLM 生成 GCJP 代码
        ↓
verify_gcjp_code(code)
        ↓
如果失败 → 提取 VerificationReport
        ↓
按错误类型构造修复 prompt
        ↓
LLM 输出新版 GCJP
        ↓
再次验证
        ↓
直到通过或达到最大轮数（建议 max_rounds=3）
```

### 2.3 修复 Prompt 分类构造

不同错误类型使用不同的修复 prompt 模板：

**L1 安全检查失败：**

```json
{
  "error_type": "SAFETY_CHECK_FAILED",
  "violations": [
    {
      "type": "DISALLOWED_IMPORT",
      "line": 2,
      "source": "import os",
      "fix_suggestion": "只允许 from gcjp.mission_graph import TaskGraphBuilder"
    }
  ],
  "original_code": "..."
}
```

**L1 执行失败：**

```json
{
  "error_type": "EXECUTION_FAILED",
  "api_error": {
    "code": "MISSING_TARGET_TASK",
    "api": "add_dependency",
    "actual": "T3",
    "expected": { "available_task_ids": ["T1", "T2"] },
    "hint": "先用 add_task() 注册该任务，或将 target 改为已注册任务名。"
  },
  "source_context": "> 12 | g.add_dependency('T1', 'T3', relation='sequence')"
}
```

**L3 Z3 UNSAT（使用阶段 0 的降噪输出）：**

```json
{
  "error_type": "Z3_UNSAT",
  "unsat_core_semantic": [
    "time_window_auto_T3_deadline_30min",
    "physical_feasibility_T1_fly_to_mark6"
  ],
  "attribution": [
    "任务链最短耗时 37.3 分钟超出 30 分钟截止时间，建议放宽 deadline 或减少中间任务"
  ],
  "original_code": "..."
}
```

### 2.4 评价指标

| 指标 | 含义 |
|------|------|
| `repair_success_rate` | 修复后最终通过验证的比率 |
| `avg_repair_rounds` | 平均修复轮数 |
| `max_repair_rounds` | 最大修复轮数（应 ≤ 3） |
| `unfixed_error_types` | 哪些错误类型始终修不好 |
| `error_transition_matrix` | 修复后错误类型如何变化（是否引入新错误） |
| `first_pass + repair_rate` | 首次生成 + 修复后的综合通过率 |

### 2.5 决策点

- 如果 `first_pass + repair_rate` ≥ 70%：核心研究假设成立，进入后续阶段
- 如果修复后通过率提升不显著（< 10%）：检查反馈信息是否足够；考虑在 prompt 中显式给出修复策略（而不只是错误描述）
- 如果 UNSAT 场景修复率远低于 L1 场景：说明归因降噪还需要进一步细化

> **为后续 few-shot、评测、微调积累样本**

>
> 可与阶段 1-2 并行

### 3.1 需要新增的文件

```

tools/
  build_seed_dataset.py       # 从 demo 提取种子数据
  validate_seed_dataset.py    # 用 VerificationPipeline 验证每条数据
  mutate_seed_dataset.py      # 模板扩展
```

### 3.2 每条数据格式

```json
{
  "sample_id": "seed_001",
  "input_type": "structured_task_description",
  "instruction": "fleet_1 first recon hq_1, then strike hq_2",
  "context": {
    "segment_id": "seg_001",
    "assigned_actors": ["fleet_1"],
    "available_actions": ["reconnaissance", "strike"],
    "scenario_id": "scenario_simple"
  },
  "target_gcjp": "from gcjp.mission_graph import TaskGraphBuilder\n...",
  "expected_verification": {
    "result": "sat",
    "nodes": [
      {"task_id": "t1_recon_hq1", "actor": "fleet_1", "action": "reconnaissance", "target": "hq_1"},
      {"task_id": "t2_strike_hq2", "actor": "fleet_1", "action": "strike", "target": "hq_2"}
    ],
    "edges": [
      {"source": "t1_recon_hq1", "target": "t2_strike_hq2", "relation": "sequence"}
    ],
    "constraint_types": ["time_order", "resource"]
  },
  "tags": ["single_actor", "sequence", "sat"]
}
```

与原方案的区别：`expected_verification` 从简单的 `nodes` 列表改为完整的图结构描述（含 actor/action/target 和边信息），支持后续做 Node-F1 / Edge-F1 评估。

### 3.3 种子数据来源

第一批直接从现有 demo 提取：

| demo | 覆盖模式 | 预期结果 |
|------|---------|---------|
| demo_03 | 4集群复杂场景 | SAT |
| demo_06 | solo段正例 | SAT |
| demo_08 | 多 actor parallel | SAT |
| demo_09 | sync + barrier | SAT |
| demo_02 | 资源超限 | UNSAT |
| demo_05 | 物理不可行 | UNSAT |
| demo_10 | 条件触发资源冲突 | UNSAT |
| demo_11 | 能力不匹配 | UNSAT |

### 3.4 模板扩展（关键改进）

**在数据结构层面做替换，而非文本层面。** 流程如下：

```
1. 解析种子 GCJP 代码 → 提取 TaskNode / DependencyEdge / Constraint 列表
2. 在数据结构层面做替换：
   - actor: fleet_1 → fleet_2 / fleet_3 / fleet_4
   - target: hq_1 → hq_2 / hq_3 / mark6 / mark7
   - action: reconnaissance → track / jam / strike / standby
   - relation: sequence → sync / barrier / condition_trigger
   - 参数: tolerance / ammo / energy / deadline
3. 联动更新关联字段：
   - action 变更时同步更新 ammo_cost / energy_cost / required_capability / duration_lb
   - 参考 configs/action_templates.yaml 中的模板参数
4. 从更新后的数据结构重新生成 GCJP 代码字符串
5. 对每条生成的样本执行 VerificationPipeline 验证
6. 验证不通过的样本自动丢弃并记录原因
```

目标：从 8 条种子扩展到 50-100 条已验证样本。

## 阶段 4：JSON → LLM → GCJP 对比实验

> **验证结构化任务计划能否稳定转成 GCJP**
>
> 预计周期：1周
>
> 前置依赖：阶段 1

### 4.1 需要新增的文件

```
experiments/
  exp_03_json_to_gcjp.py

prompts/
  json_to_gcjp_prompt.md

demos/
  demo_16_json_to_gcjp_generation.py
```

### 4.2 实验设计

```
同一份 task_plan JSON
        ↓                          ↓
task_plan_loader.py          LLM 生成 GCJP 代码
        ↓                          ↓
BuiltGraph (reference)       BuiltGraph (prediction)
        ↓                          ↓
                   结构对比
```

### 4.3 对比维度

| 维度 | 比较方式 |
|------|---------|
| 节点数量 | `len(ref.nodes) == len(pred.nodes)` |
| task_id 集合 | 集合交集 / 差集 |
| actor / action / target | 逐节点对比 |
| 边关系 | `(source, target, relation)` 三元组集合比较 |
| 约束集合 | 按 `constraint_type` 分类比较 |
| SAT/UNSAT 结果 | 是否一致 |

### 4.4 定位说明

这个阶段是**基线对照**而非核心实验。它标定 LLM 在最简单场景（完全结构化输入）下的准确率上界。后续阶段逐步增加输入模糊度（结构化描述 → 半结构化 → 纯自然语言），观察准确率衰减曲线。

> **从单段验证推进到多段拼装与全局验证**

### 5.2 跨段依赖声明格式（先于实现定义 Schema）

```json
{
  "cross_dependencies": [
    {
      "source_segment": "seg_recon",
      "source_exit_node": "T2_recon_complete",
      "target_segment": "seg_strike",
      "target_entry_node": "T1_begin_strike",
      "relation": "sequence",
      "sync_tolerance": null,
      "resource_handoff": {
        "energy_kwh_remaining": 35.0,
        "ammo_remaining": 1
      }
    }
  ]
}
```

这个 Schema 与研究计划中的接口契约格式保持一致，后续接入指挥 Agent 时不需要重构。

### 5.3 最小功能清单

```
1. 节点 ID 加 segment 前缀（seg_recon::T1 → 避免跨段冲突）
2. 合并 nodes / edges / constraints 到全局 NetworkX DAG
3. 根据 cross_dependencies 添加跨段 time_order / barrier / handoff 边
4. 沿拓扑顺序推导资源状态（上游消耗 → 下游初始值递减）
5. 输出全局 BuiltGraph
6. 调用 VerificationPipeline.verify_graph(global_built_graph)
```

### 5.4 验证方式

demo_17 手动构造两个简单的 BuiltGraph（各 2-3 个节点），通过 `cross_dependencies` 声明跨段 sequence 依赖，整合后验证全局图 SAT。再构造一个资源跨段超限的场景，验证整合后 UNSAT 且 unsat core 指向正确的跨段约束。

## 阶段 6：物理可行性自动注入

> **让系统自动根据环境坐标推导飞行时间约束**
>
> 预计周期：3-5天

### 6.1 需要修改的文件

```
gcjp/task_plan_loader.py      # 在加载 JSON 时自动注入
gcjp/environment_model.py     # 已有距离计算，需增加自动注入接口
```

### 6.2 自动注入逻辑

在 `task_plan_loader.py` 的加载流程中，对每个任务节点：

```
1. 确定任务执行位置（target → resolve_location()）
2. 确定前一个任务的执行位置（或 actor 初始位置）
3. 调用 euclidean_distance_km() 计算距离
4. 根据 actor 速度估算最短飞行时间
5. 调用 builder.add_physical_feasibility_constraint() 注入
```

### 6.3 验证方式

修改 demo_01 使其加载环境配置，对比自动注入前后的约束数量和 Z3 结果。构造一个距离远但时间窗紧的场景，验证自动注入的 physical_feasibility 约束能导致 UNSAT。

## 阶段 7：Layer 4 语义反向校验

> **检查任务图是否忠实表达原始任务语义**
>
> 预计周期：1周

### 7.1 需要新增的文件

```
verifier/
  semantic_checker.py         # Layer 4 实现

demos/
  demo_18_semantic_reverse_check.py
```

### 7.2 第一版：确定性比对

输入：原始任务描述（结构化 JSON）+ LLM 生成的 BuiltGraph

比对维度：

| 维度 | 检查方式 |
|------|---------|
| 任务覆盖率 | 原始描述中的每个任务是否在图中有对应节点 |
| actor 一致性 | 任务是否分配给了正确的执行者 |
| action 一致性 | 任务动作类型是否正确 |
| target 一致性 | 任务目标是否正确 |
| relation 一致性 | 任务间关系是否与原始描述匹配 |
| 多余节点检测 | 图中是否有原始描述中不存在的任务 |

输出：`SemanticCheckResult`，包含逐项比对结果和总体 `semantic_fidelity_score`。

### 7.3 后续版本（暂不实现）

用 LLM 将 BuiltGraph 反向生成自然语言摘要，与原始指令做语义相似度比对。这需要额外的 LLM 调用成本，留到微调阶段再考虑。

## 当前暂不优先做的事情

以下内容在核心闭环验证通过之前不应启动：

- 大规模训练 / GRPO 强化学习
- 多 Agent 微调（指挥 Agent + 规划 Agent 分别微调）
- 完整人机交互界面
- 态势摘要与候选反制计划生成
- 复杂 GUI / Web 界面
- 大规模 API 扩展

原因：核心研究假设（"LLM 能否稳定生成合法 GCJP 并通过结构化反馈修复"）尚未验证。在此之前做训练或复杂上层系统，是在未验证的假设上堆叠工程。

## 里程碑与决策点

| 里程碑 | 通过条件 | 失败降级 |
|-------|---------|---------|
| **M-0** | UNSAT 场景的 `unsat_core_semantic` 不含框架约束 | 调整过滤规则 |
| **M-1** | 强模型简单样例 `first_pass_rate` ≥ 50% | 简化 API 或改为"LLM 填模板参数"模式 |
| **M-2** | `first_pass + repair_rate` ≥ 70% | 增强反馈信息；考虑显式修复策略 |
| **M-3** | 种子数据集 ≥ 50 条且全部通过验证 | 增加模板扩展规则 |
| **M-4** | JSON→GCJP 节点/边一致率 ≥ 80% | 检查 prompt 或简化对比维度 |
| **M-5** | 两段整合后全局验证 SAT 且跨段 UNSAT 归因正确 | 修正 integrator 逻辑 |
