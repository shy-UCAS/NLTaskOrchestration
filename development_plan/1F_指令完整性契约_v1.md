# 1F 指令完整性契约 v1（Completeness Contract）

> **单一可推导源**。1F 的 prompt 规则、dev/eval 数据集的 gold 标签、评测指标三者**必须**从本契约派生；任何一处与本契约冲突，以契约为准、改那一处，而不是反过来调标签或调模型。

---

## 1. 目的与适用范围

1F 判断一条**真实指挥官自然语言指令**的**作战语义**是否足以直接交给 GCJP 生成阶段，无需操作员介入。

- `complete` = 下游规划器**仅凭文本 + 已声明约定**即可唯一推进，无需任何猜测或人工消歧。
- `incomplete` = 存在缺失、多义或被推迟到运行时/操作员的决策 → **触发 human-in-the-loop 澄清**。

设计立场：**human-in-the-loop 是安全机制**。猜错的下游代价（能力冲突、误毁伤、错配资源）远大于多问操作员一句的代价。因此边界向"存疑即 incomplete"倾斜（见 §7）。

---

## 2. 完整性谓词（核心）

> 一条指令为 `complete`，**当且仅当**下列六维**全部**可用「指令文本（+ 澄清记录）+ §4 已声明约定集」**唯一确定**。
> 任一维出现 **缺失 / 多值 / 推迟决策** → `incomplete`，并在 `missing_fields`、`ambiguities` 标出对应维。

判定对每个 task 以及任务间结构逐一施加；任一 task 任一维不达标即整条 `incomplete`。

---

## 3. 六维判定

| 维 | complete 要求 | 触发 incomplete 的形态（按原则，不靠穷举词表） | field |
|---|---|---|---|
| **actor** | 具体已声明 fleet ID（`fleet_1`..`fleet_12`） | 集合/数量化指代："主力""两个编队""几个编队""所有可用编队" | `assigned_actors` |
| **action** | 经 §4 约定集解析到**唯一**标准动作 | 动词映射到 ≥2 个动作、或不在约定集内（无法唯一归一） | `action` |
| **target** | 具体已声明标识（`area_/target_/site_/node_/point_/defense_/corridor_/...` + 名） | 描述性/相对指代且无 ID："前沿通信节点""高价值目标""北侧区域""指挥所附近""发现的目标"。**例外**：链式任务继承前文唯一已指定目标 | `target` |
| **relation**（≥2 任务时） | 每对任务关系唯一落在 {sequence, parallel, sync, conditional, fork, join} | 选择/推迟关系："或""都可以""根据情况""自行决定"；或纯并列枚举而无任何关系线索 | `relation` |
| **condition**（动作带触发时） | 条件**可机器判定**（见 §5） | 主观/不可测意图："时机成熟""被吸引后""现场判断" | `condition` |
| **split/merge** | 拆分：给出**组数 + 每组 action+target**；合并：给出**参与者 + 会合点 + 合并后任务** | 模糊规模/分配："拆成若干小组""抽调一部分""从多方向" | `split_assignment` |

## 4. 已声明约定集（Declared Convention Set）—— "可解析"的唯一定义处

"能否唯一确定 action"由**这一份封闭映射**定义。**不在表内 = 不可解析 = incomplete**。这是整个契约最关键、也是此前最易漂移的部分。

### 4.1 标准动作词表（封闭）
`reconnaissance` / `strike` / `jam` / `intercept` / `track` / `standby` / `fly_to` / `rendezvous` / `breakthrough`

### 4.2 动词 → 动作 映射（封闭；只有列出的形态可解析）

| 原文形态 | → action |
|---|---|
| 侦察、搜索、标定目标位置、确认目标位置 | `reconnaissance` |
| 打击、攻击、精确打击、摧毁、**火力压制** | `strike` |
| **电子干扰、电子压制、压制通信、压制雷达** | `jam` |
| 拦截、反制移动目标 | `intercept` |
| 持续跟踪、尾随监视 | `track` |
| 待命、保持、预备支援 | `standby` |
| 出发、前往、接近、推进到某点 | `fly_to` |
| 会合、集结、同步到达 | `rendezvous` |
| **佯攻突防、突防、突破防线** | `breakthrough` |

### 4.3 明确判为**不可解析**（→ incomplete, field=action）的形态（示范，非穷举）

- **裸"压制"**（无"电子/通信/雷达/火力"限定）：歧义于 `jam`(能量、非毁伤) vs `strike`(弹药、毁伤)，二者能力/消耗/后果完全不同 → **必须 incomplete**。
- **裸"佯动"**（无"突防/突破"）：可对应 breakthrough/recon/jam/standby 多种 → incomplete。
- "协同行动""处理""支援一下""执行任务"等泛动作。

> **裁决原则**：约定集**只声明哪些可解析**；凡未列入者，模型/标注者一律按"不可解析 → incomplete"处理，**不需要、也不应该**在 prompt 里枚举上述负面短语（见 §8 反泄漏）。

---

## 5. 条件可判定性（condition）

带触发条件的动作，其条件必须引用**可观测、可机器判定的状态**之一：

- **位置/地理围栏**："进入 corridor_X 后""离开 area_X 后"
- **数量阈值**："超过 N 架""红方出动 ≥ N"
- **目标确认事件**："发现/确认 target_X 后"
- **具名前置任务完成**："t1 完成后""突破 defense_X 后"

反之，**主观或不可测意图**（"时机成熟""红方力量被吸引""敌方注意力分散""现场判断"）→ `incomplete`，field=`condition`。

---

## 6. 1F 范围外（**不得**因其缺失判 incomplete）

以下由 1G 阶段从 `configs/action_templates.yaml` + `configs/capability_model.yaml` 注入，**指挥官原文不会提供，1F 也不向其索要**：

- `duration_lb`、`energy_cost`、`ammo_cost`
- 资源上限（弹药/能量上限）
- 能力可行性（fleet 是否真能执行该动作 —— 该校验属 1G）

---

## 7. 安全默认（平局规则）

> 若一个**审慎读者**能对任一维构造出 **≥2 种站得住脚的解读**，且文本未经 §4 约定集消歧 → 判 `incomplete`。

存疑一律偏向 incomplete。这是与 human-in-the-loop 一致的兜底，也是 §2 谓词的平局裁决器。

---

## 8. 三件产物的一致性义务（防漂移 / 防过拟合）

1. **Prompt = 契约的正向投影**：prompt 的动作表逐字等于 §4 约定集（不多不少）；条件规则等于 §5；六维要求等于 §3。
2. **反泄漏 / 反过拟合**：prompt **只声明"什么可解析"（正向约定集）**；**禁止**枚举 eval 用作 incomplete 测试项的具体含糊短语（如"前沿通信节点""北侧区域"）。incomplete 的识别必须靠原则（不在约定集 / 不可机器判定 / 多值），从而 eval 可用**任意新含糊措辞**而模型仍能凭原则判出。
3. **标签机械派生**：每条 gold 标签由 §2 谓词机械套用得出，禁止逐条凭感觉标注；标注时记录"触发 incomplete 的维"。
4. **指标可归因**：false-complete 必须按"漏判的维"(actor/action/target/relation/condition/split) 分桶；不得让某桶因数据形态而恒为 0。
5. **正负对照覆盖**：每个含糊维至少配一对"表层相近、标签相反"的样本（complete vs incomplete），用于测"判别"而非"模式匹配"。

---

## 9. 判定流程清单（标注者与模型共用）

对每条指令：

1. **逐 task**：actor 是具体 fleet？action 经 §4 唯一可解析？target 是具体 ID？
2. **任务间**：≥2 任务时，每对关系是否唯一确定（§3 relation）？
3. **条件**：每个触发条件是否可机器判定（§5）？
4. **拆分/合并**：是否给全组数+动作+目标 / 参与者+会合点+后续任务（§3 split/merge）？
5. **范围**：缺的是不是 §6 的系统参数？是 → **不**计入 incomplete。
6. 全部通过 → `complete`；否则 → `incomplete` + 标出失败维。存疑见 §7。

## 11. 判别器架构：LLM 提议 + 确定性裁决（neuro-symbolic）

**设计原则**：LLM 负责它擅长的——把脏自然语言解析成候选结构（分词、识别 actor/action/target、切任务、抽关系/条件短语）；契约的硬规则（§3/§4/§5）由**确定性代码**裁决。即 **LLM 提议，代码裁决**。约定从 prompt 搬进代码后，模型不再靠"背 prompt"得分——这是 §8.2 反泄漏义务的工程化。

裁决层是**单向收紧**：只把 `complete` 改判为 `incomplete`，绝不把 `incomplete` 升级为 `complete`（与 agent 现有一致性不变量同向），因此不引入新的错误方向。

### 11.1 分层数据流

```
raw_instruction (+ clarification_history)
        │
        ▼
┌──────────────────────────────────────────────┐
│ L1  LLM 提议层                                  │  agents/instruction_normalizer_agent.py
│   prompt 渲染 + LLM 调用 + extract_json_object  │  （prompt 由 action_lexicon 生成）
│   → 候选 resolved_fields / status               │
└──────────────────────────────────────────────┘
        │ 候选结构
        ▼
┌──────────────────────────────────────────────┐
│ L2  确定性裁决层（契约即代码）                   │  agents/instruction_validators.py（新）
│   a. 约定集解析   §4 → action                   │  configs/action_lexicon.yaml（新）
│   b. 实体 ID/注册表 §3 → actor/target           │  复用 validate 工具的 fleet 校验逻辑
│   c. 关系/schema   §3 → relation                │  已在 _complete_output_is_semantically_ready
│   d. 条件可判定性  §5 → condition               │  模板匹配
│   任一不达标 → status=incomplete + 标维（单向收紧）│
└──────────────────────────────────────────────┘
        │ 裁决后 status
        ├── incomplete ──▶ 澄清闭环（human-in-the-loop）──▶ 回 L1
        ▼ complete
┌──────────────────────────────────────────────┐
│ L3  验证 oracle（可选 / 先用于离线评测）          │  verifier/pipeline.py, gcjp/*
│   试建任务图 / 跑 SMT；complete 却 UNSAT → 反标 L1 假阳 │
└──────────────────────────────────────────────┘
        │ 通过
        ▼
   进入 1G GCJP 生成
```

### 11.2 各层职责与落点

| 层 | 做什么 | 杀哪类错误 | 接入现有代码 |
|---|---|---|---|
| **L1 LLM 提议** | 解析 NL → 候选结构 | 解析/切分错误 | [agents/instruction_normalizer_agent.py](../agents/instruction_normalizer_agent.py) `normalize()` |
| **L2a 约定集解析** | verb→action 闭集查表，命中 0 或 ≥2 即不可解析 | 压制/佯动等多义动作 | 新 `configs/action_lexicon.yaml` + `resolve_action()` |
| **L2b 实体校验** | actor 匹配 `fleet_\d+` 且 ∈ capability_model；target ∈ ID 文法/注册表 | 泛指 target、非法 actor | 复用 [tools/validate_instruction_normalization_dataset.py](../tools/validate_instruction_normalization_dataset.py) 的 fleet 校验，作用于模型输出 |
| **L2c 关系/schema** | ≥2 任务必有 relation；字段类型校验 | 多任务缺关系、字段缺失 | 已在 `_complete_output_is_semantically_ready` |
| **L2d 条件可判定性** | 条件短语匹配可测模板（围栏/阈值/确认/具名前置） | 027 类不可测条件 | 新增模板匹配（LLM 抽短语 + 代码判模板） |
| **L3 验证 oracle** | complete 试建图/跑 SMT，UNSAT 反标 | false-complete | [verifier/pipeline.py](../verifier/pipeline.py)、[gcjp/constraint_templates.py](../gcjp/constraint_templates.py) |

### 11.3 集成姿态：后置校验 vs MCP 工具调用

- **后置确定性管线（主力）**：L2 像现有不变量一样在 agent 内跑，零额外 token、可复现、可单测。
- **MCP / function tool（二阶段）**：把 `resolve_action(span)` / `is_known_entity(id)` / `can_fleet_do(fleet, action)` 暴露为工具，供 LLM 推理中主动调用核对。代价是多轮往返、更慢更贵；**仅当 L1 连候选都抽错时才值得**。

### 11.4 实施次第

1. **（先）约定集解析器 + `action_lexicon.yaml`**：prompt 动作表从它生成（§8.1 的"逐字一致"变成代码保证），`resolve_action` 做裁决。
2. **（先）实体 ID 文法 / 注册表**：复用 validate 工具逻辑，作用于模型输出。
3. **（已部分有）关系/schema 校验**：保持。
4. **（中）条件可判定性模板匹配**。
5. **（后）SMT oracle 回灌**：先用于离线评测，作 false-complete 检测。
6. **（二阶段）MCP 工具化**。

### 11.5 边界与风险（强约束）

1. **1F 能力无关**：能力交叉核对（L2 可选）只作**预警**，不作判别依据；真正的可行性裁决留 1G/SMT，否则 1F 会偷偷依赖能力模型、越界。
2. **L2 单向收紧**：只 complete→incomplete，绝不反向（同现有不变量），不引入新错误方向。
3. **检索/few-shot 泄漏**：若未来引入检索增强，检索库必须与 eval 集**严格隔离**，否则又退化为"查表"。
