# 1F 指令规范化评测重构 —— 疑问与解决记录

> 本文档以**指挥/研究者提出的疑问**为主线，记录每个疑问采用了什么方法、解决了什么问题、落地为哪些产物与证据。配套文档：[1F 指令完整性契约 v1](1F_指令完整性契约_v1.md)。

## 背景

起点是一次 `exp_01f_instruction_normalization` 的运行分析（目标：压低 `false_complete_rate`）。在排查过程中，由研究者连续提出若干疑问，逐层把问题从"指标调优"推进到"评测设计本身的正确性"，最终落到**为 1F 建立一份可推导的完整性契约**。

## 疑问一：改进方案必须是「通用规则」，不能针对这批数据做专门微调

**你的要求**
> 给出的方法必须是通用的规则，而不是针对这一批数据做的专门微调 prompt。

**采用的方法**
- 用**与具体措辞无关的原则**而非词表：未充分确定（underdetermination）、Schema 类型匹配、任务关系显式化。
- 在 **agent 代码层**加一致性不变量：`status=complete` 但 `missing_fields`/`ambiguities` 非空 → 强制 `incomplete`。该判断只看 JSON 字段是否非空，**不依赖任何中英文关键词**，天然通用。
- 设"通用性自证"三约束：示例不复用失败措辞、原则文本不点名场景词、代码不变量与措辞无关。
- 加审计指标 `model_self_consistency_rate` / `invariant_override_rate`，用于区分"prompt 真让模型学会判别" vs "靠代码兜底"。

**解决了什么**
- 改进不绑定具体样本；后续即便换措辞也应生效。
- 这条"通用 vs 过拟合"的标准成为后续所有决策的准绳，并最终写进契约 §8.2（反泄漏/反过拟合）。

**产物**：agent 一致性不变量 + 审计字段（保留至今）；指标 `model_self_consistency_rate`/`invariant_override_rate`。

## 疑问二：raw_instruction 不该包含弹药/能量消耗与上限；007 的「同时」无所指

**你的疑问**
> 指挥官的指令怎么可能告诉你消耗多少发弹药、能量消耗多少？还写能量/弹药上限？这些上限不是已经定义在 `capability_model.yaml` 里了吗？
> 007 里的"同时"是跟哪句话对应上的？

**采用的方法**
- **重新定位 1F 职责**：1F 只判作战语义（谁/做什么/对哪个目标/任务关系/条件触发），数值参数（duration/energy/ammo/资源上限）下放到 1G，由 `action_templates.yaml` + `capability_model.yaml` 注入。
- **数据集重构**：raw 改为真实指挥官口吻，删除所有人工数值；007 改写为两个显式任务 + 收口句"以上两项任务同时执行"，让"同时"有明确所指。
- **用校验工具固化约束**：`validate_instruction_normalization_dataset.py` 增 `ARTIFICIAL_PARAMETER_PHRASES`，在 CI 级别拒绝含"能量消耗/弹药消耗/能量上限/弹药上限/资源上限"的指令。
- **取证核对**：grep 全集确认无伪参数残留；逐条核对 actor 均存在于 `capability_model.yaml`（fleet_1..12）。

**解决了什么**
- raw 回归真实指挥语言；能力/消耗的唯一真源回到 `capability_model.yaml`。
- 评测从"伪参数补全"纠正为"语义完整性判定"，与系统分层一致。

**产物**：语义化 dev/eval 数据集；`exp_01f` 的 `SEMANTIC_TASK_REQUIRED_FIELDS`（actor/action/target）；校验工具的伪参数禁令。

## 疑问三：Prompt 本身泄露了测试数据

**你的疑问**
> prompt 本身泄露测试数据。

**采用的方法**
- 取证：grep 比对发现 prompt 逐字列出了 eval 的含糊短语（"前沿通信节点""高价值目标""北侧区域""协同行动""根据情况安排先后""拆成若干小组"等），且示例 2/3 直接用了 eval 措辞。
- 用契约 §8.2 给出**结构性解法**：prompt **只声明"什么可解析"（正向约定集）**；incomplete 一律靠原则判定（不在约定集 / 不可机器判定 / 多值），**不枚举负面短语**。
- 据此改写：actor/target/relation/split 四处枚举短语 → 原则表述；示例 2/3 → `demo_` 中性措辞。
- 验证：grep 确认 prompt 内已无任何 eval incomplete 短语；校验工具的 prompt-leakage 检查通过。

**解决了什么**
- 消除"模型靠查表得分"的过拟合：prompt 内列出的负面词全部通过、未列出的就出错——这一对照恰好暴露了泄漏，去词后 eval 可用任意新措辞检验真判别。
- status_accuracy 从此才能反映"真语义判别"而非"短语命中"。

**产物**：去泄漏后的 [instruction_normalization_prompt.md](../prompts/instruction_normalization_prompt.md)；契约 §8.2 反泄漏义务。

## 疑问四（最核心）：歧义指令被标 complete，违背 human-in-the-loop 设计——数据集是否偏离了原设计？

**你的疑问**
> 设计本意是"存在明显歧义则需操作员介入(human-in-the-loop)"。但 eval 里像"未明确是电子干扰还是火力打击、无法归一"的歧义指令，却被标成 `complete`。这个数据集测试是不是偏离了原本的设计？

**采用的方法**
- **读实际模型输出取证**，而非空谈：
  - eval_003 "压制 site_orion"：模型判 incomplete，理由"未明确 jam 还是 strike"——**比金标签更严谨**。
  - eval_010 "压制 site_victor"：同样裸"压制"，模型却归一为 jam 判 complete——**同词不同判，数据集自相矛盾**。
- **引入关键区分**："约定可解" vs "真 underspecified"——不能把任何字面多义都判 incomplete（会过度触发），但也不能把真歧义放行。
- **定位根因**：项目缺一份显式的"完整性契约"，complete/incomplete 边界是逐条凭感觉画的，故出现自相矛盾。
- **起草《完整性契约 v1》**作为单一可推导源：六维谓词 + 已声明约定集 + 条件可判定性 + 安全默认（存疑即 incomplete）+ 三件产物一致性义务。
- **按契约裁定并修数据**：裸"压制"=不可解析，按"安全默认"应 incomplete；据决定 A 改写为"电子压制"以保留 complete 覆盖（003/010/015/018/039 + dev_010）。

**解决了什么**
- 确认你的判断成立：对这些 case 而言数据集确实偏离了 human-in-the-loop 设计。
- 把"逐条凭感觉标注"升级为"从契约机械派生标签"，根除同类自相矛盾。
- 顺带统一裁定了 R3 三个错误的真因（027 条件不可测、036 佯动、003 压制），证明它们都是 **prompt↔数据集规格冲突**而非模型能力问题。

**产物**：[1F 指令完整性契约 v1](1F_指令完整性契约_v1.md)；压制规范化后的数据集；prompt 动作表对齐 + 新增 §5 条件可判定性 + 示例 4/5。
