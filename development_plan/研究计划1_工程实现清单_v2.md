# 研究计划1 工程实现清单（修订版v2）

> 基于v4.4精简大纲，综合两版方案优势，按假设验证优先级重组

---

## 总体原则

1. **先确定性底座，再接入LLM**
2. **先结构化输入→代码生成，再自然语言→结构化**
3. **先单Agent跑通，再拆多Agent**
4. **先v4.3核心闭环，再v4.4态势增强**
5. **每一层都有明确的通过条件和失败降级方案**

---

# 第一层：确定性底座（纯工程，不涉及LLM）

> **目标：** 搭建可手动灌入数据、跑通"拼装→验证→输出调度"的完整管道
>
> **预计周期：** 3-4周

### 1.1 JSON Schema 定义

- [ ] 标准化任务计划格式 schema
  - [ ] 含 tasks 列表（task_id / actor / action / target / condition / expected_output）
  - [ ] 含 relations 列表（source / target / type / condition）
  - [ ] 含 participants 列表
  - [ ] 含 ambiguities 和 missing_fields 字段
- [ ] 集群能力模型格式 schema（含 composition + fleet_constraints）
- [ ] 动作模板库格式 schema（每个动作的 min_duration / resource_cost / requires / parallelizable）
- [ ] 环境配置格式 schema（目标点坐标、威胁区、会合点、禁飞区）
- [ ] 指挥Agent分解输出 schema（segments / topology / segment_interfaces）
- [ ] 规划Agent GCJP 代码 META 头格式（SEGMENT_META / CONTRACT_FULFILLMENT / RESOURCE_STATE_AT_EXIT）
- [ ] 态势结构化摘要格式 schema
- [ ] 候选反制措施格式 schema
- [ ] 验证报告格式 schema（四层结果 + unsat core 归因）
- [ ] schema 校验工具函数（jsonschema / pydantic）

### 1.2 静态配置文件

- [ ] capability_model.yaml（2-4个集群的能力定义，含异构组成）
- [ ] action_templates.yaml（reconnaissance / track / intercept / strike / fly_to / jam / breakthrough / standby）
- [ ] environment_config.yaml（示例场景的目标点坐标）
- [ ] tactical_knowledge.yaml（3-5条典型反制模式）

### 1.3 GCJP 受限 API

- [ ] `TaskGraphBuilder` 类：基于 NetworkX DiGraph
  - [ ] `add_task(task_id, actor, action, target, duration_lb, required_capability, energy_cost, ...)`
  - [ ] `add_dependency(source, target, relation, condition=None)`
  - [ ] `add_constraint(constraint_type, params, source_label)`
  - [ ] `add_resource_constraint(actor, resource_type, max_value)`
  - [ ] `declare_contract_fulfillment(interface_id, exit_node, resource_state)`
  - [ ] `declare_resource_state(actor, remaining_ammo, remaining_range, ...)`
- [ ] Z3 约束模板注册器
  - [ ] 时间顺序：`start_j >= end_i`
  - [ ] 持续时间：`end_i = start_i + duration_i`，`dur_i >= min_duration`
  - [ ] 时间窗：`earliest_i <= start_i <= latest_i`
  - [ ] 同步：`abs(start_i - start_j) <= sync_tolerance`
  - [ ] 资源：`sum(cost) <= budget`
  - [ ] 能力匹配：`required ⊆ capability`（先Python检查，保留Z3接口）
  - [ ] 物理可行性：`dur >= dist / speed`
- [ ] 命名约束追踪机制（`solver.assert_and_track`）
- [ ] API 白名单校验器

### 1.4 四层递进验证管道

- [ ] **第一层 - 代码执行验证**
  - [ ] subprocess 沙箱 + 超时控制
  - [ ] 语法检查 + 运行时错误捕获
  - [ ] 非法 API 调用检测
- [ ] **第二层 - 图结构验证**
  - [ ] DAG合法性 / 连通性 / 节点覆盖 / 孤立节点
  - [ ] 拓扑排序 / 关键路径
- [ ] **第三层 - Z3 约束验证**
  - [ ] solver 构建与求解
  - [ ] SAT → 调度方案提取
  - [ ] UNSAT → unsat core → 归因映射
  - [ ] 超时处理
- [ ] **第四层 - 语义反向校验（预留接口）**
- [ ] 验证报告生成器（结构化JSON + 可读文本）

### 1.5 确定性整合模块

- [ ] 局部 GCJP 代码加载（解析META头）
- [ ] 命名空间统一
- [ ] 契约一致性校验（guarantees vs assumptions）
- [ ] 跨段连接边自动生成
- [ ] 资源状态确定性推导
- [ ] 全局 GCJP 代码输出

### 1.6 手写端到端样例

- [ ] 样例1：2集群纯solo（3-4节点，线性依赖）
- [ ] 样例2：2集群 + 1个会合点（同步约束）
- [ ] 样例3：3集群 + 联合段（coalition）
- [ ] 样例4：4集群多阶段协同（15+节点）
- [ ] 样例5：含不可行约束的反例（UNSAT路径）
- [ ] 每个样例：NL指令 / 标准化任务计划JSON / 分解JSON / 各段GCJP代码 / 预期结果
- [ ] 集成测试全部通过

### 1.7 辅助工具

- [ ] 任务图可视化（graphviz）
- [ ] 甘特图渲染
- [ ] unsat core 可读化报告

### 1.8 中央编排器骨架

- [ ] `pipeline_controller.py`：定义完整流程接口
- [ ] 支持模块级跳过/替换
- [ ] 统一日志和计时

**✅ M1：** 5个手写样例全部通过验证管道，含1个UNSAT正确归因

# 第二层：分步假设验证（强模型 Few-shot）

> **目标：** 分步验证LLM能否完成各环节，精确定位瓶颈
>
> **预计周期：** 2-3周

### 2.1 后半段验证：结构化任务计划 → GCJP代码

> 输入已结构化的JSON，只测代码生成能力

- [ ] 编写规划Agent System Prompt（GCJP API文档 + 输出格式）
- [ ] 强模型few-shot生成，从样例1到样例4递增复杂度
- [ ] 灌入验证管道，统计代码可执行率 / 图结构合法率 / Z3通过率
- [ ] 记录典型失败模式
- [ ] 必要时调整GCJP API设计

### 2.2 前半段验证：NL → 标准化任务计划

> 输入NL指令，只测结构化抽取能力

- [ ] 编写指令解析 System Prompt（输出schema + 歧义检测）
- [ ] 强模型few-shot生成
- [ ] 校验主体/动作/关系抽取准确性
- [ ] 记录典型失败模式

### 2.3 串联验证

- [ ] 2.2输出 → 2.1输入，串联运行
- [ ] 端到端Z3通过率
- [ ] 失败归因：NL理解错还是代码生成错

### 2.4 指挥Agent分解验证

- [ ] 编写指挥Agent System Prompt
- [ ] 测试分解准确性（覆盖/联合段识别/契约完整/拓扑）

### 2.5 可行性报告

- [ ] 各环节成功率
- [ ] 失败模式分类
- [ ] Go/No-go决策

**✅ M2：** 强模型端到端Z3通过率 ≥ 60%

**❌ 若 < 40%：** 简化API；改JSON→确定性翻译；降低复杂度
