# 研究计划1：面向无人机集群的自然语言到GCJP任务图生成与SMT闭环验证方法（v4.6）

---

## 一、研究目标

面向无人机集群对抗场景，研究从自然语言作战指令到结构化、可验证、可执行任务有向无环图（DAG）的自动构建方法。

本研究以指挥员自然语言指令为核心输入，结合敌方态势描述、我方集群能力、环境约束、动作模板库与战术知识库，充分利用大模型对结构化代码的预训练优势，使LLM直接生成图—约束联合代码表示（Graph-Constraint Joint Python，GCJP）。GCJP代码在受限Python环境中执行后构建任务图与形式化约束，并通过NetworkX图结构检查与SMT/Z3约束验证，判断任务图在时序一致性、资源可行性、能力匹配、协同关系与抽象物理可行性方面是否满足要求。

本研究聚焦任务规划层，输出“谁在什么时候去哪里做什么”的抽象任务图、关键依赖关系、调度骨架、资源分配结果与下游执行输入参数，不直接生成连续轨迹、底层控制律或飞控指令。

本研究的核心目标不是构建一个以标准化JSON为运行时核心的任务解析流水线，而是形成如下运行时主线：

```text
自然语言指令 + 态势/能力/环境/模板/战术上下文
        ↓
LLM直接生成GCJP代码
        ↓
代码沙箱执行并构建任务图
        ↓
图结构检查 + SMT/Z3约束验证
        ↓
冲突归因与定向修复
        ↓
可验证任务DAG与调度方案
```

其中，标准化任务计划JSON仅作为训练标注、过程奖励和工程调试的辅助表示，不作为系统运行时必经步骤。

## 二、问题定义

### 2.1 输入

系统输入包括以下内容：

1. **指挥员自然语言作战指令**：描述任务目标、执行对象、时间要求、协同行为和约束条件；
2. **敌方态势描述**：包括敌方目标位置、运动方向、行为模式、威胁等级、预计到达时间和可能意图；
3. **我方无人机集群能力模型**：包括平台类型、可用数量、载荷能力、侦察能力、打击能力、续航、通信范围、剩余资源等；
4. **环境配置**：包括目标区域、防御边界、威胁区、会合点、禁飞区、起降点、巡逻区和关键保护目标；
5. **动作模板库**：包括侦察、跟踪、压制、拦截、打击、待机、会合、转场、掩护、巡逻等动作模板；
6. **战术知识库**：包括典型反制模式、任务优先级规则、资源使用规则、协同处置建议和安全边界；
7. **人机交互确认信息**：当指令模糊、目标冲突或候选反制措施需要确认时，由指挥员给出确认、修改或拒绝结果。

### 2.2 输出

系统输出包括以下内容：

1. **GCJP图—约束联合代码**：由LLM直接生成，调用受限API描述任务节点、任务依赖与形式化约束；
2. **可验证任务DAG**：由GCJP代码执行后构建，表示任务节点、边关系和任务属性；
3. **形式化约束集合**：包括时间、资源、能力、同步、条件触发与抽象物理可行性约束；
4. **验证报告**：包括代码执行结果、DAG合法性、SMT/Z3可满足性、冲突归因和语义反向校验结果；
5. **调度与执行输入参数**：包括任务开始/结束时间、关键路径、资源分配、执行主体分配和下游执行模块所需参数。

### 2.3 任务图表示

任务图采用有向图表示，节点、边和约束共同构成任务计划。

- **节点**表示原子任务事件，包含任务编号、执行主体、动作类型、目标区域、任务参数、资源需求、能力需求、执行条件和持续时间下界；
- **边**表示任务间依赖关系，包括顺序、并行、同步、屏障、条件触发、信息传递和资源接续；
- **约束**表示任务执行必须满足的形式化条件，包括时间顺序、时间窗、资源预算、能力匹配、同步容差、条件触发和抽象物理可达性；
- **验证结果**用于判断任务图是否满足形式化一致性、资源可行性、协同正确性和语义忠实性。

## 三、核心挑战

1. **多主体跨句语义理解**：自然语言指令中常包含多个无人机集群、多个敌方目标和多个任务阶段，需要识别省略、指代、隐含事件和跨句依赖。
2. **态势依赖型任务补全**：原始指令可能只给出高层意图，需要结合敌方态势、我方资源和战术知识补全任务目标、触发条件、处置优先级和候选行动。
3. **协同时序关系建模**：无人机集群任务涉及顺序执行、并行行动、同步会合、条件触发、全局屏障和联合行动，需要统一转化为图边和形式化约束。
4. **端到端GCJP代码生成正确性**：LLM需要从自然语言和上下文直接生成语法正确、API调用合法、图结构完整、约束一致的GCJP代码。
5. **多Agent局部生成一致性**：复杂场景下多个规划Agent并行生成局部GCJP代码，必须保证节点命名、上下游条件、资源状态、跨段依赖和约束来源一致。
6. **任务层抽象物理可行性**：虽然不生成连续轨迹，但任务图仍需考虑飞行时间下界、航程预算、弹药预算、能力匹配、会合可达性和同步时间窗。
7. **形式化验证与冲突归因**：当验证失败时，需要定位冲突来自具体任务节点、依赖边、资源约束、能力约束、时间窗还是GCJP代码片段。
8. **人机交互式任务确认**：在任务意图模糊、候选措施冲突或风险较高时，需要将候选反制措施提交给指挥员确认，再融合进最终任务意图。

## 四、方法概述

### 4.1 总体思路

提出一种态势增强的GCJP代码生成与SMT闭环验证框架。核心思想是：

> 利用大模型的代码生成能力，从自然语言指令和结构化上下文直接生成可执行、可检查、可验证的GCJP代码；再利用代码沙箱、图结构检查、SMT/Z3求解和语义反向校验构成验证闭环，以验证结果驱动冲突归因和定向修复。

运行时主线流程如下：

```text
NL指令 + 态势/能力/环境/模板/战术上下文
        ↓
上下文构造与态势增强
        ↓
LLM直接生成GCJP候选代码
        ↓
受限Python沙箱执行
        ↓
构建NetworkX任务图 + Z3约束集合
        ↓
代码执行验证 → 图结构验证 → SMT约束验证 → 语义反向校验
        ↓
冲突归因与定向修复
        ↓
可验证任务DAG + 调度方案 + 执行输入参数
```

整体步骤如下：

1. 将自然语言指令、敌方态势、我方能力、环境配置、动作模板和战术知识组织为LLM可读的结构化上下文；
2. 在指令模糊、目标冲突或处置策略不明确时，系统从态势描述中提取威胁摘要，生成候选反制措施，并通过人机确认确定最终任务意图；
3. LLM直接生成GCJP代码，代码调用受限Python API构建任务图并编码形式化约束；
4. 对于复杂多集群场景，由编排Agent生成局部任务包，多个同构规划Agent并行生成局部GCJP代码，再由确定性整合模块构建全局任务图；
5. 验证Agent执行代码执行检查、图结构验证、SMT/Z3验证和语义反向校验；
6. 当验证失败时，系统基于异常信息、图结构错误、Z3 unsat core和语义偏差位置进行冲突归因，并将问题路由到对应规划Agent或整合模块进行定向修复。

### 4.2 结构化上下文与结构化任务计划的区别

本研究中的“结构化上下文”不等同于“结构化任务计划”。

- **结构化上下文**是LLM生成GCJP代码时使用的外部条件集合，包括态势摘要、能力模型、环境配置、动作模板库和战术知识库；
- **结构化任务计划JSON**是训练和调试阶段使用的辅助标注，用于构造高质量样本、计算节点覆盖奖励和定位错误来源；
- 运行时系统不要求先生成标准化任务计划JSON，也不将JSON作为任务执行或验证的核心表示；
- 系统唯一可执行、可检查、可验证的运行时表示是GCJP代码及其执行后构建的任务图和约束集合。

### 4.3 标准化任务计划JSON的定位

标准化任务计划JSON不作为运行时必经步骤，仅在以下场景中使用：

1. **训练数据标注层**：构建`(NL指令, 上下文) → GCJP代码`配对数据时，标准化任务计划作为中间标注，辅助检查GCJP代码是否覆盖任务语义；
2. **过程奖励信号**：在强化学习阶段，检查LLM生成的GCJP代码是否覆盖标准化计划中的任务节点、依赖关系和约束条件；
3. **工程调试层**：当端到端生成失败时，对照标准化任务计划判断失败来自自然语言理解、任务分解、代码生成还是约束编码；
4. **Gold标注辅助层**：在构建评估集时，标准化任务计划可辅助人工形成Gold任务DAG，但不参与运行时推理。

## 五、总体框架

### 5.1 上下文构造与态势增强层

该层负责将原始自然语言指令与多源外部信息整合为LLM生成GCJP代码所需的结构化上下文。

主要功能包括：

- 指令模糊检测与冲突检测；
- 敌方态势摘要提取；
- 候选反制措施生成；
- 人机确认信息融合；
- 集群能力模型注入；
- 环境配置注入；
- 动作模板库和战术知识库注入；
- 最终任务意图确定。

### 5.2 GCJP代码生成层

该层由LLM直接从自然语言指令和结构化上下文生成GCJP代码。

主要功能包括：

- 从NL指令直接输出调用受限API的Python代码；
- 生成任务节点、任务依赖和形式化约束；
- 生成约束来源追踪信息，便于后续冲突归因；
- 对复杂场景启用多Agent局部生成机制；
- 在代码生成前进行粗粒度可行性筛查。

### 5.3 复杂场景多Agent分解层

对于涉及多集群、多阶段、多目标和多约束的复杂场景，启用多Agent分解生成机制。

需要特别说明的是：多Agent分解生成的“任务包”不作为最终任务表示，也不替代GCJP。任务包的作用仅是缩小单次代码生成上下文、约束局部规划Agent的输入边界，并辅助保持跨段一致性。最终系统仍以GCJP代码和全局任务图作为唯一可执行表示。

该层主要包括：

1. **编排Agent**：识别任务阶段、参与集群、敌方目标、局部任务段和跨段依赖，生成局部任务包；
2. **规划Agent**：基于局部任务包生成局部GCJP代码；
3. **一致性约束**：规定节点命名、资源状态、输入输出条件和跨段依赖接口；
4. **同构模型调用**：编排Agent和规划Agent是同一基座模型的不同角色调用实例，不是独立训练的多个模型。

### 5.4 确定性整合层

该层负责将多个局部GCJP片段拼装为全局任务图和全局约束集合。

主要功能包括：

- 局部代码合并；
- 节点命名规范化；
- 跨段依赖连接；
- 资源状态推导；
- 重复节点检测；
- 不兼容约束检测；
- 全局任务图构造；
- 全局约束集合生成。

### 5.5 验证修复层

该层负责对GCJP代码、任务图和形式化约束进行递进式验证，并在失败时执行冲突归因与定向修复。

主要功能包括：

- Python代码执行验证；
- 受限API调用验证；
- NetworkX图结构验证；
- SMT/Z3约束验证；
- 语义反向校验；
- unsat core冲突归因；
- 局部代码修复；
- 任务包重分解；
- 人工确认回退。

## 六、关键技术路线

### 6.1 输入建模

构建统一输入表示体系，将多源信息组织为LLM可处理的上下文。

输入表示包括：

- 自然语言指令表示；
- 态势摘要表示；
- 集群能力模型表示；
- 环境配置表示；
- 动作模板表示；
- 战术知识表示；
- 人机确认表示。

示例上下文结构如下：

```json
{
  "instruction": "命令侦察分队跟踪东北方向快速突防目标，打击分队在其进入Z1后实施拦截。",
  "situation_summary": {...},
  "friendly_capabilities": {...},
  "environment": {...},
  "action_templates": [...],
  "tactical_knowledge": [...],
  "human_confirmation": {...}
}
```

### 6.2 态势摘要提取

从敌方态势描述中提取与任务规划相关的关键字段。

字段包括：

- 敌方集群编号；
- 子群划分；
- 当前空间位置；
- 运动方向；
- 行为模式；
- 可能意图；
- 威胁目标；
- 预计到达时间；
- 威胁等级；
- 处置优先级；
- 态势发展趋势。

示例：

```json
{
  "enemy_group": "E1",
  "behavior": "rapid_breakthrough",
  "approach_direction": "northeast",
  "threat_target": "P1",
  "estimated_arrival_time": 8,
  "threat_level": "high",
  "priority": 1,
  "possible_intent": "breakthrough"
}
```

### 6.3 候选反制措施生成

基于态势摘要、我方资源状态、环境约束和战术知识库，生成候选反制措施清单。

每个候选措施包含：

- 措施编号；
- 适用态势；
- 反制目标；
- 建议动作；
- 推荐执行主体；
- 触发条件；
- 资源需求；
- 预期效果；
- 风险说明；
- 推荐理由；
- 可验证约束提示。

候选反制措施不直接进入最终任务图，必须经过规则过滤、资源检查和必要的人机确认后，才能融合进最终任务意图。

### 6.4 人机交互确认

在以下情况下触发人机确认：

- 自然语言指令存在歧义；
- 态势摘要与指令目标存在冲突；
- 多个候选反制措施均可行但优先级不明确；
- 候选措施涉及高风险资源使用；
- SMT验证失败且自动修复需要改变任务目标或执行主体。

确认方式包括：

- 全部采纳；
- 部分采纳；
- 修改后采纳；
- 暂不采纳；
- 重新生成；
- 补充新指令。

确认结果被写入LLM上下文，作为GCJP代码生成的约束条件之一。

### 6.5 GCJP受限API设计

GCJP采用受限Python API表示任务图和形式化约束。与直接字符串形式的约束表达不同，本研究采用结构化约束API，以提高静态检查能力、变量一致性和冲突归因能力。

核心API包括：

```python
# 创建任务图构建器
g = TaskGraphBuilder()

# 添加任务节点
g.add_task(
    task_id="T1",
    actor="scout_team_A",
    action="track",
    target="E1",
    region="Z0",
    duration_lb=5,
    required_capabilities=["recon"],
    resources={"energy": 10},
    source="nl_sentence_1"
)

# 添加依赖边
g.add_dependency(
    source="T1",
    target="T3",
    relation="condition_trigger",
    condition="E1 enters Z1",
    source_ref="dep_from_instruction_1"
)

# 添加时间顺序约束
g.add_time_order(
    before="T1",
    after="T3",
    relation="end_before_start",
    source="time_order_T1_T3"
)

# 添加资源预算约束
g.add_resource_limit(
    actor="scout_team_A",
    resource="energy",
    max_value=100,
    used_by=["T1"],
    source="resource_scout_A_energy"
)

# 添加能力匹配约束
g.add_capability_requirement(
    task_id="T1",
    actor="scout_team_A",
    capability="recon",
    source="capability_T1_recon"
)

# 添加同步约束
g.add_sync_constraint(
    tasks=["T4", "T5"],
    sync_type="start_within",
    tolerance=2,
    source="sync_T4_T5"
)
```

### 6.6 GCJP约束模板库

构建面向无人机任务图的SMT约束模板库。

主要约束包括：

1. **时间顺序约束**：`start_j >= end_i`；
2. **持续时间约束**：`end_i = start_i + duration_i`；
3. **时间窗约束**：`earliest_i <= start_i <= latest_i`；
4. **同步约束**：`abs(start_i - start_j) <= sync_tolerance`；
5. **资源约束**：`sum(resource_cost(actor, tasks)) <= resource_budget(actor)`；
6. **能力匹配约束**：`required_capability(task) ⊆ capability(actor)`；
7. **条件触发约束**：`trigger_condition_i → start_j >= trigger_time_i`；
8. **会合可达性约束**：`travel_time(actor, rendezvous_point) <= available_time`；
9. **抽象物理可行性约束**：`flight_time_lb <= available_time_window`；
10. **资源接续约束**：同一执行主体连续任务之间必须满足转场时间和资源剩余条件。

每条约束都应携带`source`字段，以支持unsat core回溯、错误定位和局部修复。

### 6.7 端到端GCJP代码生成

LLM接收完整上下文后直接生成GCJP代码。

生成内容包括：

- 任务节点定义；
- 任务边定义；
- 节点属性；
- 边属性；
- 时间约束；
- 资源约束；
- 能力约束；
- 同步约束；
- 条件触发约束；
- 抽象物理可行性约束；
- 约束来源追踪信息。

生成过程不要求模型显式输出标准化任务计划JSON。可在训练阶段选择性引入简短任务理解摘要作为CoT引导，但运行时最终提交给验证器的是GCJP代码。

### 6.8 多Agent分解生成机制

复杂场景启用多Agent分解生成机制，流程如下：

1. 编排Agent接收完整上下文，识别任务阶段、参与集群、目标对象、局部任务段和跨段依赖；
2. 编排Agent生成局部任务包，任务包仅用于约束规划Agent的输入边界，不作为最终任务表示；
3. 多个规划Agent并行生成局部GCJP代码；
4. 规划Agent在代码生成前进行粗粒度可行性筛查，包括动作能力匹配、航程预算、弹药或载荷、时间窗可行性、会合点可达性、上下游条件兼容性和资源竞争检查；
5. 确定性整合模块合并局部GCJP代码，统一节点命名，补全跨段依赖，推导资源状态，形成全局任务图和全局约束集合；
6. 全局验证失败时，系统将错误路由到对应规划Agent、编排Agent或整合模块进行定向修复。

### 6.9 递进式验证机制

验证Agent执行四层验证。

#### 6.9.1 代码执行验证

检查内容包括：

- Python语法正确性；
- 受限API调用合法性；
- 禁止文件读写、网络访问、系统调用等危险操作；
- 任务节点和约束对象是否可解析；
- 代码执行是否能成功生成任务图对象。

#### 6.9.2 图结构验证

检查内容包括：

- DAG合法性；
- 节点ID唯一性；
- 边两端节点存在性；
- 孤立节点；
- 循环依赖；
- 拓扑排序可行性；
- 关键路径可计算性；
- 节点、边和任务属性覆盖情况。

#### 6.9.3 SMT/Z3统一验证

检查内容包括：

- 时序一致性；
- 持续时间合法性；
- 时间窗可满足性；
- 资源预算可满足性；
- 能力匹配一致性；
- 同步可达性；
- 条件触发一致性；
- 抽象物理可行性。

#### 6.9.4 语义反向校验

从任务图反向重构关键信息描述，与原始自然语言指令和人机确认结果进行语义一致性对比。

检查内容包括：

- 是否遗漏关键任务；
- 是否增加无依据任务；
- 执行主体是否一致；
- 目标对象是否一致；
- 时序关系是否一致；
- 条件触发是否一致；
- 候选反制措施是否按确认结果融合。

### 6.10 冲突归因与定向修复

冲突类型包括：

- 代码执行错误；
- 非法API调用；
- 节点遗漏；
- 冗余节点；
- 依赖关系错误；
- 时间冲突；
- 资源冲突；
- 能力冲突；
- 同步冲突；
- 抽象物理不可达；
- 语义偏移。

修复方式包括：

- 修复API调用；
- 补充缺失节点；
- 删除冗余节点；
- 修改依赖边类型；
- 调整时间窗；
- 重分配执行主体；
- 调整资源使用；
- 增加同步等待节点；
- 重新生成局部GCJP代码；
- 回退编排Agent重新分解；
- 触发人工确认。

当Z3返回UNSAT时，系统基于unsat core定位相关约束的`source`字段，并进一步映射到任务节点、依赖边、GCJP代码片段或规划Agent输出，从而执行定向修复，而不是整体重新生成。

### 7.2 Gold标注来源

端到端结构准确性以人工审查后的Gold任务DAG和Gold约束集合为参照。标准化任务计划JSON仅用于辅助构造Gold标注、辅助错误诊断和计算过程奖励，不参与运行时推理。

Gold标注构建流程如下：

1. 人工或强模型根据自然语言指令和上下文生成初始标准化任务计划；
2. 人工审查任务节点、依赖关系和约束条件；
3. 将审查后的任务计划转写为GCJP代码；
4. 执行GCJP代码生成任务DAG和约束集合；
5. 通过NetworkX与Z3验证后，形成Gold任务DAG和Gold约束集合；
6. 对存在争议的样本记录歧义说明和人工确认结果。

### 7.3 标准化任务计划JSON的训练价值

标准化任务计划JSON虽不出现在运行时主线中，但在数据集中具有重要价值。

- **SFT监督辅助**：为`NL → GCJP`提供中间对照，便于构建高质量配对数据；
- **过程奖励**：用于检查GCJP代码是否覆盖任务节点、依赖边和约束条件；
- **错误诊断**：用于区分失败来自NL理解错误、任务分解错误、代码生成错误还是约束编码错误；
- **弱监督中间表示**：可用于训练阶段的隐式任务理解能力学习，但不要求模型运行时显式生成JSON。

### 7.4 数据类型

数据集包含以下类型：

1. 明确指令样本；
2. 模糊指令样本；
3. 态势增强样本；
4. 人机确认样本；
5. 多集群协同样本；
6. 多阶段任务样本；
7. 资源冲突样本；
8. 能力不匹配样本；
9. 同步冲突样本；
10. 失败修复样本。

### 7.5 数据构建流程

1. 人工设计30—50条种子任务；
2. 按复杂度Level 1—5扩展自然语言指令；
3. 强模型辅助生成态势描述、候选反制措施和初始GCJP代码；
4. 人工审查标准化任务计划和GCJP代码；
5. 自动执行代码检查、图结构检查和SMT/Z3验证；
6. 收集失败样本并构造修复轨迹；
7. 形成训练集、验证集和测试集；
8. 对测试集进行人工复核，保证Gold任务DAG和Gold约束集合质量。

## 八、训练策略

### 8.1 总体策略

采用“领域适应—端到端GCJP代码生成特化—验证驱动强化优化”的渐进式训练策略。

训练目标是让模型逐步获得以下能力：

- 理解无人机集群任务语义；
- 理解态势、能力和环境约束；
- 熟悉GCJP受限API；
- 能够从自然语言和上下文直接生成GCJP代码；
- 能够根据验证反馈进行定向修复。

### 8.2 领域适应阶段

增强模型对无人机任务语义和GCJP API的基础理解能力。

训练内容包括：

- 多主体事件识别；
- 动作类型识别；
- 目标区域识别；
- 时序关系理解；
- 协同关系理解；
- 资源与能力约束理解；
- GCJP API用法学习；
- 无人机领域术语学习。

### 8.3 端到端GCJP代码生成特化阶段

核心训练任务为：

```text
(NL指令, 态势/能力/环境/模板/战术上下文) → GCJP代码
```

训练时可以利用标准化任务计划JSON作为中间质量检查，但模型输入仅为NL指令和上下文，模型目标输出为GCJP代码，不要求显式生成中间JSON。

可选择性引入简短CoT训练：让模型在生成代码前输出简短任务理解摘要，再生成GCJP代码。该摘要不强制匹配标准化任务计划格式，只作为生成前的语义整理。

### 8.4 多Agent相关训练任务

针对复杂场景，设置两个附加训练任务：

1. **编排Agent训练**：从NL指令和上下文生成任务分解、局部任务包和协同时序骨架；
2. **规划Agent训练**：从局部任务包生成局部GCJP代码。

需要强调的是，编排Agent任务包是生成辅助，不是最终任务表示；最终监督仍落在GCJP代码、任务DAG和验证通过率上。

### 8.5 验证驱动强化优化阶段

利用自动验证信号构造奖励函数。

奖励包括：

- GCJP代码可执行性；
- 受限API合法性；
- DAG合法性；
- SMT可满足性；
- 资源约束一致性；
- 能力匹配一致性；
- 同步约束一致性；
- 语义反向校验一致性；
- 修复后通过率；
- 平均修复轮数降低；
- 节点覆盖率；
- 边覆盖率；
- 约束覆盖率。

### 8.6 多Agent联合优化

以统一验证器作为集中式评价器，优化目标包括：

- 编排Agent分解合理性；
- 规划Agent局部生成正确性；
- 跨Agent输出一致性；
- 确定性整合成功率；
- 全局任务图验证通过率；
- 平均修复轮数；
- 多集群复杂场景可扩展性。

## 十、预期创新点

1. **自然语言到GCJP的端到端代码生成范式**：利用大模型代码预训练先验，从自然语言指令和上下文直接生成包含任务图与形式化约束的GCJP代码，避免将标准化JSON作为运行时必经中间步骤。
2. **图—约束联合代码表示GCJP**：设计受限Python API，在统一代码表示中同时表达NetworkX任务图结构和Z3形式化约束，使任务计划具备可执行、可检查、可验证特性。
3. **结构化GCJP约束API与来源追踪机制**：用结构化API替代自由字符串约束，增强静态检查、变量一致性、unsat core回溯和局部修复能力。
4. **态势增强上下文构造机制**：将敌方态势摘要、候选反制措施、我方能力模型、环境配置、动作模板和战术知识融合为LLM生成上下文，提高任务补全和歧义消解能力。
5. **递进式验证与SMT闭环修复**：构建“代码执行—图结构—SMT/Z3—语义反向”的四层验证管道，基于验证反馈进行冲突归因和定向修复。
6. **复杂场景多Agent局部GCJP生成机制**：对于大规模多集群任务，由编排Agent生成局部任务包，多规划Agent并行生成局部GCJP，再由确定性模块全局拼装。
7. **面向无人机任务图的SMT约束模板库**：覆盖时序、资源、能力、同步、条件触发、会合可达性和抽象物理可行性约束。
8. **验证驱动强化优化方法**：以代码可执行性、DAG合法性、SMT可满足性、语义忠实性和修复效率作为奖励信号，优化端到端GCJP生成质量。
9. **MilTaskGraph-v4.6数据集**：构建面向无人机集群对抗任务的NL-to-GCJP配对数据集，包含Gold任务DAG、Gold约束集合、标准化任务计划训练标注和修复轨迹。

## 十一、技术风险与应对

### 11.1 LLM端到端GCJP生成质量不足

- **风险**：LLM从自然语言直接生成GCJP代码时，首次通过率可能较低。
- **应对**：使用递进式验证与闭环修复兜底；引入few-shot示例、API文档和简短CoT引导；若失败率过高，可临时降级为`NL → 标准化JSON → GCJP`流水线作为工程备份。

### 11.2 结构化上下文过长或噪声过多

- **风险**：态势、能力、环境和战术知识注入过多，可能干扰LLM生成。
- **应对**：设计上下文压缩与字段筛选机制，仅注入与当前任务相关的能力、区域、动作模板和战术规则。

### 11.3 态势描述不稳定

- **风险**：上游态势描述可能存在遗漏、误判或表达不一致。
- **应对**：使用固定态势摘要字段，增加威胁等级、处置优先级和关键目标一致性检查。

### 11.4 候选反制措施过度自由生成

- **风险**：LLM可能生成不符合资源条件、能力约束或任务规则的建议。
- **应对**：使用候选措施模板、动作模板库、资源能力检查和人机确认进行过滤。

### 11.5 多Agent输出不一致

- **风险**：不同规划Agent可能生成冲突节点、重复节点或不兼容约束。
- **应对**：采用统一命名规则、任务包边界约束、确定性整合模块和全局SMT验证。

### 11.6 GCJP代码安全风险

- **风险**：生成代码可能存在语法错误、非法API调用或危险操作。
- **应对**：使用受限API白名单、静态检查、代码沙箱和执行权限限制。

### 11.7 SMT约束不可满足

- **风险**：任务计划在时间、资源、能力、同步或可达性方面不可行。
- **应对**：通过unsat core定位冲突约束，执行时间窗调整、执行主体重分配、资源使用调整或人工确认回退。

### 11.8 数据集构建成本较高

- **风险**：完整标注GCJP代码、Gold任务DAG、态势摘要和修复轨迹成本较高。
- **应对**：采用人工种子样本、模板扩展、强模型辅助生成、自动验证筛选、人工抽查和失败样本回收机制。

## 十二、阶段性实施计划

### 阶段1：GCJP表示与验证底座构建

- 设计GCJP受限API；
- 设计结构化约束API；
- 定义任务DAG节点与边格式；

### 阶段2：GCJP生成可行性拆解验证

该阶段用于拆分“代码生成能力”和“自然语言理解能力”的错误来源。

#### 实验2.1：JSON → GCJP

- 输入人工审查后的标准化任务计划JSON；
- 使用强模型生成GCJP代码；
- 验证GCJP API、Prompt、代码沙箱和验证器是否可用；
- 统计代码可执行率、DAG合法率和Z3通过率。

#### 实验2.2：NL → GCJP

- 输入自然语言指令和结构化上下文；
- 使用强模型直接生成GCJP代码；
- 与实验2.1比较，定位失败来自NL理解还是代码生成；
- 输出端到端生成可行性报告；
- 决定是否引入CoT引导或临时降级方案。

### 阶段3：态势增强与人机确认

- 构建态势摘要字段体系；
- 实现态势描述到结构化摘要转换；
- 构建候选反制措施模板；
- 引入战术知识库和资源过滤机制；
- 实现人机确认流程。

### 阶段4：种子数据集构建

- 设计30—50条种子指令；
- 按Level 1—5设置复杂度等级；
- 强模型生成初始GCJP代码；
- 人工修正形成Gold标注；
- 构建标准化任务计划JSON作为训练标注层；
- 模板扩展至200—300条；
- 自动验证筛选并人工抽查；
- 格式化为JSONL并划分训练、验证和测试集。

### 阶段5：模型微调与验证驱动强化优化

- 选择7B级模型进行最小SFT验证；
- 进行领域适应训练；
- 进行端到端GCJP代码生成特化训练；
- 引入验证驱动强化优化；
- 与强模型和规则基线进行对比。

### 阶段6：多Agent分解与全局拼装

- 实现编排Agent任务分解；
- 实现规划Agent局部GCJP生成；
- 实现确定性整合模块；
- 实现跨段依赖连接和资源状态推导；
- 在复杂多集群样本上验证可扩展性。

### 阶段7：实验评估与论文整理

- 完成端到端代码生成实验；
- 完成对比实验；
- 完成消融实验；
- 完成多Agent复杂场景实验；
- 重点评估运行时是否加入标准化JSON中间步骤的影响；
- 整理论文方法、实验和结果分析。

## 十四、版本修改说明

相较于v4.5，本版本主要修改如下：

1. 将标题进一步聚焦为“自然语言到GCJP任务图生成与SMT闭环验证”，突出`NL → GCJP → Verification`主线；
2. 明确区分“结构化上下文”和“结构化任务计划JSON”，避免将上下文注入误解为运行时中间任务表示；
3. 明确标准化任务计划JSON仅用于训练标注、过程奖励、工程调试和Gold辅助构建，不作为运行时必经步骤；
4. 补充多Agent任务包定位，强调任务包只是局部生成辅助，不替代GCJP；
5. 将GCJP约束表达从自由字符串形式改为结构化API形式，增强静态检查和冲突归因能力；
6. 补充Gold任务DAG和Gold约束集合的来源，明确Node-F1、Edge-F1、Constraint-F1等指标的参照对象；
7. 将阶段2改为“GCJP生成可行性拆解验证”，区分`JSON → GCJP`与`NL → GCJP`两个子实验；
8. 增加结构化约束API、Constraint-F1、多Agent复杂场景指标和运行时中间步骤对比消融。
